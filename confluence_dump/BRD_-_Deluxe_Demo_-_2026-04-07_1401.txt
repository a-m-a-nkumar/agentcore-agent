Merchant Portal Feature Enhancement &ndash; Accept &amp; Sell Control Center --- 1. Document Overview **Project Name:** Merchant Portal Feature Enhancement &ndash; Accept &amp; Sell Control Center **Document Version:** 1.0 **Date:** February 18, 2026 **Prepared By:** Product Management Team **Document Status:** Draft for Review **Project Summary:** Evolution of the Merchant Portal into a unified &quot;Accept &amp; Sell&quot; control center featuring secure checkout optimization, real-time transaction dashboard, and insights reporting module to improve merchant conversion, reduce disputes, and drive platform stickiness. --- 2. Purpose This BRD defines requirements for enhancing the Merchant Portal to: Transform the portal from a configuration console to an active merchant sales enablement tool Provide secure, optimized checkout experiences with risk-based authentication Deliver real-time transaction visibility and reporting capabilities Surface actionable, predictive insights to help merchants improve business outcomes Increase merchant engagement, reduce churn, and improve competitive positioning --- 3. Background / Context **Current Pain Points:** Checkout configuration is complex and buried in advanced settings Transaction reporting lags by 24 hours due to batch ETL processing Merchants rely on manual exports and reconciliation No actionable insights or predictive analytics available Portal experience not competitive with market alternatives Consistent global merchant feedback requesting better checkout, reporting, and insights Mid-market merchant demands for secure branded checkout, real-time reporting, and predictive analytics Competitive pressure requiring differentiation through data-driven merchant value Opportunity to leverage existing transaction data across regions **Business Triggers:** **Market Context:** Mid-market merchants (North America, LATAM, APAC) seeking operational efficiency Industry trend toward embedded, customizable checkout experiences Growing merchant expectation for real-time data and AI-driven recommendations --- 4. User Stories / Use Cases # User Story 1: Merchant Configures Secure Checkout **As a** mid-market merchant **I want to** easily configure secure checkout with optimal settings **So that** I can maximize conversion while maintaining security compliance **Acceptance:** Guided wizard walks through configuration in &lt;5 minutes System recommends 3DS settings based on my geography and risk profile I can preview checkout before going live Branding elements display correctly # User Story 2: Merchant Monitors Real-Time Performance **As a** merchant operations manager **I want to** view today's transaction performance in real-time **So that** I can quickly identify and respond to issues **Acceptance:** Dashboard shows transactions within 5 minutes of occurrence I can filter by payment method, region, and time Approval rate and decline reasons are clearly visible I receive alert when approval rate drops below threshold # User Story 3: Merchant Receives Actionable Insights **As a** merchant without a data team **I want to** receive clear recommendations to improve my payment performance **So that** I can increase revenue without hiring analysts **Acceptance:** Insights appear as notification cards on dashboard Recommendations are specific and actionable (e.g., &quot;Enable local acquiring in Brazil&quot;) I can see projected impact of implementing recommendations Insights are accurate based on my historical data # User Story 4: Finance Team Reconciles Transactions **As a** merchant finance team member **I want to** easily reconcile transactions with settlement reports **So that** I can close books accurately and quickly **Acceptance:** Transaction ID, order ID, and settlement reference clearly mapped Export functionality provides all necessary reconciliation data Multi-currency transactions show FX impact Data accuracy is 99.9%+ # Use Case: Risk-Based Authentication Flow **Actors:** Merchant, End Customer, Fraud Engine, Checkout System **Preconditions:** Merchant has enabled risk-based authentication **Main Flow:** Customer initiates checkout on merchant site Checkout system captures transaction details System calls fraud scoring engine (&lt;300ms) If risk score is LOW: Frictionless authentication flow Transaction proceeds without 3DS challenge If risk score is HIGH: Step-up authentication triggered 3DS challenge presented to customer Transaction proceeds after successful authentication Transaction result displayed to customer Merchant sees conversion metrics updated in real-time **Postconditions:** Transaction processed with appropriate authentication level, metrics updated # Use Case: Alert-Driven Issue Resolution **Actors:** Merchant, Dashboard System, Notification Service **Preconditions:** Merchant has configured alert thresholds **Main Flow:** Dashboard monitors approval rate continuously Approval rate drops 3% below threshold in Brazil region System generates alert with diagnostic information Alert delivered via email and in-portal notification Merchant logs into portal and views detailed decline breakdown System surfaces insight: &quot;Issuer declines increased 15% - consider enabling local acquiring&quot; Merchant reviews recommendation and impact projection Merchant initiates configuration change or contacts support **Postconditions:** Merchant aware of issue and has actionable path to resolution --- 10. Assumptions Fraud engine APIs can support real-time scoring under required latency (&lt;300ms) No major regulatory changes in SCA or PCI-DSS standards during development period At least 12 months of transaction data available for active merchants to generate meaningful insights Merchants willing to adopt new checkout flows without significant integration friction Current merchant base has technical capability to implement embedded SDK BNPL providers have standardized API interfaces available for integration Merchant Success teams available to support pilot program and onboarding Data science team can develop and validate predictive models within timeline Cloud infrastructure can scale to support 3x transaction volume growth Regional gateway nodes can be upgraded to support real-time data streaming Merchants have modern browsers supporting required web technologies Third-party security reviews can be completed within allocated timeline --- 11. Constraints **Budget:** Fixed fiscal year allocation; large re-architecture requires phased rollout **Technical:** Legacy architecture in certain regions (especially LATAM gateway nodes) may limit real-time data ingestion capabilities **Resource:** Engineering team already committed to tokenization upgrade project; parallel workstreams required **Timeline:** 8-10 month delivery window for competitive positioning **Compliance:** PCI-DSS review and penetration testing adds 1 month to timeline **Data Quality:** Insight accuracy dependent on historical data cleanliness; requires validation layer **Regional:** Data localization requirements in India and APAC countries limit architecture flexibility **Integration:** Third-party BNPL provider integration dependent on their API availability and security approval **Performance:** Must maintain &lt;2 second checkout latency to avoid cart abandonment impact **Scope:** Phase 1 limited to hosted and embedded checkout; headless API deferred to Phase 2 --- 12.
